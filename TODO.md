Thin adapter by design — shells out to the same `wb` binary (`current_exe`) for run/inspect/validate/resume/pending (so `run_single`'s `process::exit`-on-pause becomes a mappable exit code instead of killing the server) and reads checkpoint+pending in-process (read-only) for `get_run_events`. State mapping: checkpoint+pending = the Task store (`run_id` = checkpoint id); `pause_for_human`/`wait` → `status:"input_required"` + an `elicitation` object (message + `requestedSchema`, one property per bound var) that the client satisfies via `resume_workbook`; task status rides on the child exit code (0→completed, 42→input_required, 1→failed, 7→timeout, …). Zero new deps (`serde_json` only). End-to-end author→run→pause→resume→read verified in `tests/mcp_e2e.rs`; unit coverage in `src/mcp.rs`. Hosted HTTP/SSE transport + a true server-initiated `elicitation/create` round-trip are deferred (the CLI subprocess model can't hold a call open mid-pause).

Trust + registry / remote execution.** Wave 5 shipped the safe remote-execution piece: `wb run gh:OWNER/REPO/PATH[@REF]` and `wb run https://…md` fetch the workbook to `~/.wb/remote/<hash>.md` (via curl) and **always** trust-gate it (TOFU, #37) — a fetched remote workbook is never auto-executed; it refuses until the operator reviews the cached file and `wb trust add`s it. This pairs remote execution with the trust the roadmap requires. Cryptographic **signing** also shipped (see #37 — `wb keygen`/`sign`/`verify-sig`, ed25519): a remote workbook can be both TOFU-gated *and* signature-verified (`wb run gh:… --verify-sig --pubkey <author>`). Still open: a hosted **registry/gallery** on workbooks.dev and the atoms/flows/tasks package model — these need a backend service.
- [~] **41. `wb capture` — record a session → emit a workbook.** Shipped the command-sequence form in wave 5: `wb capture` reads commands from stdin (one per line; `#` lines become Markdown headings), runs each in a real `executor::Session`, and emits a runnable workbook — each command as a block, optionally followed (`--assert`) by an `expect` fence carrying the observed exit code + a quote-safe stdout substring. `-o <file>` or stdout; `--runtime`/`--title`/`-C`. The generated workbook validates and re-runs under `wb test`. Still open: full PTY/interactive session recording, browser-recording reuse, and MANIFEST.md-compatible output (feeds the registry, #40).
- [x] **42. `wb run --repair` — self-healing runs.** Wave 5 shipped all four actions: `wb run --repair <url> [--repair-max N]` POSTs a failed block (event/index/language/exit_code/error_type + **redacted** stdout/stderr) and applies the returned `{"action":"rerun"|"skip"|"abort"|"patch","code":…}` — `rerun` re-executes in place, `skip` continues past the failure, `abort`/unknown/unreachable stops (fail-safe), and **`patch`** runs an endpoint-supplied replacement command in place of the failed block (logged prominently — never silent — and bounded by `--repair-max`, default 3). `patch` is genuine code execution, but `--repair` is an explicit opt-in to an operator-chosen endpoint that already controls the run, and the workbook itself is arbitrary code; pair with `--verify-sig` (#37) when running untrusted workbooks. Makes unattended operation real.

- [x] **47. Reproducibility — `wb.lock` + signed run attestations.** Wave 5 shipped the lockfile half (`src/lockfile.rs`): `wb lock <file>` writes `<file>.lock` recording a sha256 per resolved step (includes expanded, so included-file edits show up), and `wb run --locked [--lockfile <path>]` refuses to run if the workbook's input identity drifted (step added/removed/edited) since the lock — the `Cargo.lock`/`package-lock.json` analogue for runbooks. Both halves (lockfile + signing) are shipped and composable.
