# Push to GitHub

Stage, commit, push changes, and open a PR. Direct pushes to `main` are blocked; all changes go through pull requests.

## Instructions

1. Run these commands in parallel to understand the current state:
   - `git status` to see staged and unstaged changes
   - `git diff --cached` to see what's currently staged
   - `git diff` to see unstaged changes
   - `git log --oneline -5` to see recent commit message style
   - `git branch --show-current` to check the current branch

2. If there are no changes at all (nothing staged, nothing unstaged): inform the user and stop.

3. If on `main`, create a new branch before committing:
   - Choose a descriptive branch name based on the changes (e.g., `fix/typo-in-footer`, `feat/add-contact-form`)
   - Run `git checkout -b <branch-name>`

4. Determine what to commit based on the current staging state:

   **Case A — Staged changes exist, and all unstaged changes are unrelated:**
   - Commit only the currently staged items as-is. Do not stage anything else.

   **Case B — No staged changes exist:**
   - Identify all changed files. Group related changes together.
   - Stage all related items and proceed.

   **Case C — Staged changes exist, and some unstaged changes are related:**
   - Stage the related unstaged items alongside the already-staged items.
   - Leave unrelated unstaged items alone.

   Use the content of the diffs (not just filenames) to judge relatedness — changes that serve the same feature, fix, or purpose are "related."

5. Run `git diff --cached` again to review the final set of staged changes.

6. Analyze the staged changes and draft a commit message:
   - Summarize the nature of the changes (new feature, bug fix, refactor, etc.)
   - Draft a concise commit message (1-2 sentences) focusing on the "why"
   - Do NOT include "Co-Authored-By" or "Generated by" lines

7. Present the commit message to the user for verification:
   - Show which files will be committed
   - If any unstaged files were left out, mention them and why
   - Show the proposed commit message
   - Ask if they want to proceed, modify the message, or cancel

8. After user approval:
   - Create the commit with the approved message
   - Push the branch to the remote: `git push -u origin <branch-name>`
   - Open a pull request against `main` using `gh pr create`
   - Use a concise PR title and include a summary of changes in the body
   - Return the PR URL to the user
   - Never use "Co-Authored-By" or "Generated by" in PRs

9. If push fails due to rejected changes, inform the user they need to pull first.

10. **NEVER push directly to `main`.** All changes must go through a PR.
